## 🟢 中規模Phase（5段階）参考テンプレート

**対象**: Phase B（プロジェクト管理）
**特性**: プロジェクト・ユーザー関連管理・権限設定
**組織規模**: 標準SubAgentパターン活用
**期間**: 5-7セッション
**レビュー**: 開始・中間・完了
**テスト重点**: 基本CRUD・関連管理・品質保証

### 段階構成（マスタープラン準拠）
```
Step 1: 調査分析（Step1共通プロセス）
Step 2-3: 基本実装段階（基本CRUD・関連管理・機能完成）
  - Step 2: プロジェクト基本CRUD実装
  - Step 3: ユーザー・プロジェクト関連管理・機能完成
Step 4-5: 品質保証段階（技術負債解消・UI/UX最適化・統合テスト）
  - Step 4: 品質改善・技術負債解消
  - Step 5: UI/UX最適化・統合テスト
```

### 組織設計原則
- **並列分析適用**: 課題発見・技術調査段階（Step1）
- **標準実装適用**: Pattern A/B活用による効率実装（Step2-3）
- **品質集中適用**: Pattern D活用による品質保証（Step4-5）

### 時間見積もり参考値
- Step1: 60-90分（分析・計画詳細化）
- Step2-3: 120-180分/Step（基本実装・関連機能）
- Step4-5: 90-120分/Step（品質改善・統合テスト）

## 📊 組織設計判断基準

### 組織継続 vs 変更の判断基準
- **作業性質**: 並列 vs 順次
- **専門性要求**: 深い専門知識 vs 統合調整
- **成果物特性**: 分析・設計 vs 実装・統合
- **課題状況**: 新規発見 vs 既知解決

### Phase規模による組織最適化
- **🟢中規模**: 標準SubAgentパターン活用・Pattern A/B/D中心
- **🟡大規模**: 専門性強化SubAgentパターン・Pattern A/B/D + 複雑統合対応
- **🔴超大規模**: 複雑Phase対応SubAgentパターン・全Pattern活用・段階的統合重視

## 📋 共通Step 1実行プロセス（全Phase共通）

### 標準実行手順
```
1. 【組織設計】 Phase特性に応じた専門チーム構成決定
2. 【仕様確認】 機能仕様書の該当セクション精読・仕様準拠監査役の任命
3. 【データベース設計書詳細確認】 該当Phase関連テーブル構造・制約・型変換の完全理解
4. 【各チーム専門分析】 並列実行による専門領域の深堀り調査（仕様準拠観点含む）
5. 【統合分析】 各チーム結果の統合・優先順位付け・実装順序決定
6. 【計画詳細化】 Step2以降の具体的作業計画・組織構成変更検討
7. 【詳細記録】 各チーム分析結果と統合結果の体系的記録（仕様準拠マトリックス含む）
```

### 必須確認事項
```
□ 【組織設計】 Phase特性に応じた専門チーム構成決定（仕様準拠監査役含む）
□ 【仕様確認】 必須実施項目:
  - 機能仕様書の該当セクション特定・精読
  - 肯定的仕様（実装すべき機能）の抽出
  - 否定的仕様（実装してはいけない機能）の抽出
  - 仕様準拠マトリックスの初期作成
□ 【データベース設計書詳細確認】 必須実施項目:
  - 該当Phase関連テーブル構造の完全理解
  - 外部キー関係・制約条件の確認
  - F#↔C#型変換に必要なデータ型確認
  - 実装対象エンティティの詳細仕様確認
  - ASP.NET Core Identity統合ポイント確認
□ 【各チーム専門分析】 標準情報カテゴリに従った詳細記録（仕様準拠観点含む）
□ 【統合分析】 チーム間連携・依存関係・実装順序の決定
□ 【計画詳細化】 Step2以降の具体的作業計画・組織構成変更検討
□ 【詳細記録】 /Doc/05_Research/Phase_XX/ 配下への体系的記録
```
