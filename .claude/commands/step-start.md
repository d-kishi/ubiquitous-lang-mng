# Step開始準備Command

**目的**: Step開始時の必須準備プロセス・組織設計を自動実行
**対象**: 全Step開始時（Step1, Step2, Step3...）

## コマンド実行内容

### 1. Step情報収集・確認（必須）
- [ ] **Phase状況確認**: `/Doc/08_Organization/Active/Phase_XX/`配下の現在Phase状況確認
- [ ] **前Step完了確認**: 前Step完了レビュー結果・成果物出力状況確認
- [ ] **Phase計画確認**: `Phase_Summary.md`から全Step構成・当該Step位置確認
- [ ] **次Step判定**: Phase計画から実行すべき次Stepの自動判定
- [ ] **🆕 関連Issue確認**: `Phase_Summary.md`からStep対応Issue番号を確認。Issue番号がある場合は`gh issue view [番号]`でIssue内容を読み込み、Step目的・背景・解決すべき課題を把握
- [ ] **🆕 Step1成果物参照準備**: `Phase_Summary.md`の「Step間成果物参照マトリックス」から当該Step必須参照ファイル特定

### 2. Step組織設計（簡素化・効率化）
- [ ] **Step特性判定**: 当該Stepの作業特性判断（分析/実装/テスト/統合/品質保証）
- [ ] **段階種別判定**: Phase B/C/D対応・段階種別の自動判定
  - **基本実装段階（1-3）**: 基本CRUD・関連機能・機能完成
  - **品質保証段階（4-6）**: 技術負債解消・UI/UX最適化・統合テスト
  - **拡張段階（7-8）**: 高度機能・外部連携・運用最適化
- [ ] **🎯 SubAgent組み合わせ選択**: 特性・段階に応じたPattern A/B/C/D/Eから選択
  - **Pattern A**: 新機能実装 (Domain→Application→Infrastructure→Web UI)
  - **Pattern B**: 機能拡張 (影響分析→実装・統合→品質保証)
  - **Pattern C**: 品質改善 (課題分析→改善実装→検証・完成)
  - **Pattern D**: 品質保証段階 (技術負債特定→品質改善→統合検証)
  - **Pattern E**: 拡張段階 (外部連携設計→拡張実装→運用準備)
- [ ] **🚀 並列実行必須**: 選択SubAgentを同一メッセージ内で複数Task実行
- [ ] **StepXX_[内容].md作成**: Step組織設計記録ファイル作成（簡素化テンプレート）
- [ ] **🆕 Step1成果物必須参照セクション追加**: 当該Step組織設計記録にStep1分析成果の必須参照リスト自動追加

### 2.3. Step実施内容の対話的詳細化（phase-start Section 1.5パターン適用・🆕）

**目的**: Step実施内容を対話的に具体化・合意形成（Phase_Summary.md概要 → 実施可能な詳細計画）

**プロセス**: Claude初期案提示 → AskUserQuestion活用 → ユーザー対話 → 合意形成 → 確定記録

#### Step 1: 実装対象の対話的具体化
- [ ] **Claude初期案提示**:
  - Phase_Summary.mdから当該Step実施内容を抽出
  - 実装対象（ファイル・機能・画面）の初期案を提示
  - 例: 「Step1は UI設計書 3.6-3.8節（ユーザー一覧・登録・編集の3画面）を実装対象と考えますが、いかがでしょうか？」

- [ ] **ユーザー対話・合意（AskUserQuestion活用）**:
  - 選択肢形式でスコープ確認・調整
  - 追加機能・削減機能の協議
  - 実現可能性・優先度の議論

- [ ] **合意・確定記録**:
  - 合意したスコープをStep組織設計ファイルに明記
  - 実装対象リスト作成（チェックボックス形式）

#### Step 2: SubAgent選択の対話的確認
- [ ] **Claude初期案提示**:
  - subagent-selection Command実行結果に基づくSubAgent選択案を提示
  - 例: 「このStepでは csharp-web-ui + e2e-test Agentの組み合わせを提案しますが、いかがでしょうか？」

- [ ] **ユーザー対話・合意（AskUserQuestion活用）**:
  - SubAgent選択の妥当性確認
  - 並列実行の可能性確認
  - 推定時間の妥当性確認

- [ ] **合意・確定記録**:
  - 合意したSubAgent構成をStep組織設計ファイルに記録
  - 並列実行計画を記録

#### Step 3: 成果物・完了基準の対話的定義
- [ ] **Claude初期案提示**:
  - Phase_Summary.mdから当該Step完了基準を抽出
  - 成果物リストを提示
  - 例: 「Step1完了時には以下の成果物が揃っていることを完了基準とします: 1) 3画面実装完了、2) E2Eテスト10シナリオPass、3) 単体テスト80%+カバレッジ。これで良いでしょうか？」

- [ ] **ユーザー対話・合意（AskUserQuestion活用）**:
  - 完了基準の妥当性確認
  - 測定指標の明確化
  - 受け入れ基準の合意

- [ ] **合意・確定記録**:
  - 合意した完了基準をStep組織設計ファイルに記録
  - 成果物チェックリスト作成

#### 最終確認: Step実施内容の検証
- [ ] **ユーザーに最終確認**:
  - 「Step完了時にどうなっているべきか、明確にイメージできていますか？」
  - 「どのファイル・機能が実装されていますか？」
  - 「どのテストがPassしていますか？」

- [ ] **曖昧さの排除**:
  - 曖昧な点があれば、計画段階で徹底的に明確化
  - 「なんとなくOK」を許容しない
  - Step開始前に100%の合意形成

**Phase A教訓の適用**:
- ❌ 曖昧な計画 → 実装に集中 → "なんとなくOK" → 対応漏れ
- ✅ 計画に集中 → ゴール明確化 → 実装は計画通り → 100%達成

## 実行後アクション

### Step準備完了時
✅ **Step開始準備完了** → ユーザー承認取得後にSubAgent並列実行・Step作業開始  
⚠️ **準備不足項目あり** → 不足項目完了・再確認実施  
❌ **前Step未完了** → 前Step完了確認・完了後再実行

### SubAgent実行開始条件
- **必須**: ユーザーによるSubAgent実行計画承認
- **確認事項**: 選択SubAgent・並列実行戦略・推定所要時間の最終承認
- **実行開始**: 承認取得後にSubAgent並列実行開始

### 次プロセス連携
- **SubAgent並列実行**: ユーザー承認後の専門作業実行
- **step-end-review**: Step完了時の包括的品質確認・レビュー
